# Synapse 設計書 v0.4 ドラフト — 動的媒体選択と有識者自動選出

## 変更概要

v0.3 からの主な進化:
- 成果物が Python コード固定 → ゴールに最適な媒体を自動選択
- エージェントが 3体固定 → Orchestrator が最適な専門家を動的に選出
- Synapse 自身が設計書を読み、自己進化するループの導入

## 第1章 動的媒体選択

### 1.1 概要

ユーザーのゴールを分析し、最適な出力媒体を Orchestrator が自動判定する。

### 1.2 媒体カタログ

| ID | 媒体 | 出力形式 | 判定キーワード例 |
|----|------|----------|-----------------|
| web | Web サイト/LP | HTML, CSS, JS | サイト, LP, ページ, Web |
| webapp | Web アプリ | React/Vue + API | アプリ, ダッシュボード, 管理画面 |
| cli | CLI ツール | Python | ツール, スクリプト, 自動化, CUI |
| api | REST API | FastAPI/Express | API, エンドポイント, サーバー |
| doc | ドキュメント | Markdown/PDF | 資料, レポート, 企画書, 議事録 |
| data | データ分析 | Python + matplotlib | 分析, グラフ, 可視化, CSV |
| mobile | モバイルアプリ | Flutter/React Native | モバイル, iOS, Android |
| game | ゲーム | Python/JS (様々) | ゲーム, 遊ぶ, プレイ |
| design | デザイン | Figma指示/SVG/CSS | デザイン, UI, モックアップ |
| writing | 文章・創作 | Markdown/テキスト | 小説, 物語, エッセイ, ブログ, 記事, シナリオ, 脚本 |

### 1.3 判定フロー

1. ユーザーがゴールを入力
2. Orchestrator がゴール文をキーワード分析 + LLM判定
3. 媒体IDを1つ選択（複数該当時は最も適切なものを選択）
4. 媒体に応じた専門家プロンプトとツールセットを読み込み
5. 以降のフローは v0.3 と同じ（Coder → Reviewer ループ）

### 1.4 Orchestrator の判定プロンプト（追加部分）

ゴールを分析し、以下の形式で出力:
- MEDIA: (媒体ID)
- EXPERTS: (必要な専門家のリスト)
- PLAN: (実装計画)

## 第2章 有識者の動的選出

### 2.1 概要

Orchestrator が Coder と Reviewer のシステムプロンプトを
タスクに応じて動的に生成する。エージェント数は増やさず、
プロンプトの中身を差し替えることでコストを据え置く。

### 2.2 専門家カタログ

| ID | 専門家 | 得意領域 | 割り当て先 |
|----|--------|----------|-----------|
| frontend | フロントエンド専門家 | HTML/CSS/JS, React, UI実装 | Coder |
| backend | バックエンド専門家 | API設計, DB, サーバー | Coder |
| fullstack | フルスタック開発者 | Web全般 | Coder |
| data_scientist | データサイエンティスト | 分析, 可視化, 統計 | Coder |
| game_dev | ゲーム開発者 | ゲームロジック, 物理演算 | Coder |
| technical_writer | テクニカルライター | 文書作成, 構成 | Coder |
| ux_designer | UXデザイナー | ユーザビリティ, 導線設計 | Reviewer |
| security_expert | セキュリティ専門家 | 脆弱性, 安全性 | Reviewer |
| qa_engineer | QAエンジニア | テスト設計, 品質管理 | Reviewer |
| performance_eng | パフォーマンス専門家 | 速度, 最適化 | Reviewer |
| novelist | 小説家・ストーリーテラー | 構成, キャラクター, 文体 | Coder |
| editor | 編集者 | 校正, 構成チェック, 読みやすさ | Reviewer |
| screenwriter | 脚本家 | 対話, シーン構成, 演出 | Coder |


### 2.3 選出フロー

1. Orchestrator がゴールと媒体から必要な専門家を2名選出
2. Coder用の専門家プロンプトを生成（例: frontend + game_dev）
3. Reviewer用の専門家プロンプトを生成（例: qa_engineer）
4. 各エージェントのシステムプロンプトに専門知識を注入
5. 以降のフローは v0.3 と同じ

### 2.4 プロンプト生成テンプレート

Coder のベースプロンプト + 専門家の知識を結合:

あなたは Synapse の Coder です。
今回のタスクでは以下の専門知識を持って実装してください:
- {expert_1の専門知識と注意点}
- {expert_2の専門知識と注意点}

### 2.5 コスト影響

- API呼び出し回数: v0.3 と同じ（3回/ラウンド）
- トークン増加: プロンプト拡張で約10-20%増
- 総コスト: 1タスクあたり $0.25-0.40（v0.3 比 +15%程度）

## 第3章 自己進化ループ

### 3.1 概要

Synapse が自分自身の設計書を読み、次のバージョンの
コードを生成する。人間は設計書のレビューと承認のみ行う。

### 3.2 進化サイクル

1. 人間が設計書 vN を作成（または Synapse が提案）
2. Synapse が設計書 vN を読み、コードを生成
3. 人間がコードをレビュー・テスト・承認
4. Synapse が実行結果を分析し、設計書 vN+1 を提案
5. 人間が vN+1 を承認 → 2に戻る

### 3.3 安全策

- 人間が各サイクルで必ず承認する（Human Sovereignty 原則の維持）
- 設計書の差分を明示し、何が変わったか透明にする
- 自己進化の範囲を制限（コア設計原則は人間のみ変更可能）
- 各バージョンを Git でタグ管理し、いつでもロールバック可能

### 3.4 ロードマップ

| Phase | 内容 | 人間の役割 |
|-------|------|-----------|
| Phase 1 (現在) | 人間が設計書を書き、Synapse がコード生成 | 設計 + レビュー |
| Phase 2 | Synapse が設計書の改善案を提案 | 承認 + 修正 |
| Phase 3 | Synapse が設計書とコードを両方生成 | 承認のみ |
| Phase 4 | Synapse が自動テスト→自動改善 | 監視のみ |

## 第4章 実装計画

### 4.1 v0.3 → v0.4 の変更点

| ファイル | 変更内容 |
|---------|---------|
| config.py | 媒体カタログ、専門家カタログの定義追加 |
| prompts.py | 動的プロンプト生成関数の追加 |
| engine.py | Orchestrator の判定結果に基づくプロンプト差替えロジック |
| tools.py | 媒体別ツールセット（HTML書き出し等）の追加 |
| ui.py | 媒体選択の表示、プレビュー機能の拡張 |

### 4.2 優先順位

1. Orchestrator の媒体判定プロンプト（最小変更で効果大）
2. 専門家プロンプトテンプレート 3-4 種類
3. engine.py のプロンプト差替えロジック
4. 自己進化ループの仕組み（Phase 2以降）

### 4.3 MVP で対応する媒体

まずは以下の3媒体から:
- cli (Python CLI) — 既存
- web (HTML/CSS/JS) — 新規
- doc (Markdown) — 新規
