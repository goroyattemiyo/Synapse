# Synapse 設計書 v1.0
## Multi-AI Agent Collaborative System — Prompt Architecture Edition

---

## 第1章 ビジョンと原則

### 1.1 Synapse とは何か

Synapse は、ユーザーが自然言語で伝えたゴールに対し、3つの役割（Orchestrator・Coder・Reviewer）が計画・実装・検証を自律的に繰り返し、完成品を返すシステムである。

v0.3 ではPythonアプリとして実装したが、v1.0 ではプラットフォーム非依存の「指示体系（プロンプトアーキテクチャ）」として再定義する。GPTs（ChatGPT）、Gem（Gemini）、Claude Projects、またはAPIベースの自作エンジンなど、任意の実行環境で動作する。

### 1.2 設計の変遷

| Version | 主な進化 | 教訓 |
|---------|---------|------|
| v0.3 | 3エージェント基本構成、ツール権限分離、Dockerサンドボックス設計 | 最小構成でも実用的な品質が出る |
| v0.4 | 動的媒体選択（10種）、専門家カタログ（13名）、自己進化ループ構想 | ビジョンは広げすぎると実装が追いつかない |
| v0.5 | 複雑度事前判定、環境プロファイル、ツール予算制、構造化ログ | 実測データ（$0.22 vs $2.94）が設計を正す |
| v0.6 | セキュリティ基盤、ベンチマーク、ユーザー介入、有識者5名レビュー | 安全性と計測可能性が公開の前提条件 |
| v1.0 | プロンプトアーキテクチャとして再定義、プラットフォーム非依存化 | アプリを作るより指示体系を磨く方が価値が高い |

### 1.3 5つの設計原則

1. **Human Sovereignty（人間主権）** — 最終承認・中断・方針変更の権限は常にユーザーにある
2. **Transparency（透明性）** — 各Stepの冒頭に役割を明示し、判断過程を可視化する
3. **Minimum Viable Agent（最小構成）** — 3役割を1つのAIが担う。エージェント数は増やさない
4. **Design for Failure（障害前提）** — 修正回数に上限を設け、無限ループを防止する
5. **Economic Viability（経済性）** — 複雑度に応じて出力量を制御し、コストを予測可能にする

## 第2章 アーキテクチャ

### 2.1 実行フロー

Copy
User Goal ↓ Step 0: 分析（複雑度・媒体・専門家を自動判定） ↓ Step 1: 【Orchestrator】計画 ↓ ← complexの場合、ここでユーザー確認を挟む Step 2: 【Coder】実装 ↓ Step 3: 【Reviewer】検証 ↓ ├─ APPROVED → Step 4: 納品 ├─ NEEDS_REVISION → Step 2 に戻る（最大2回） └─ 3回目 → APPROVED（条件付き）で納品


### 2.2 タスク複雑度の事前判定（v0.5 由来）

ローグライクダンジョン生成で$2.94を消費し4ラウンドで未承認に終わった実測から導入。

| 複雑度 | 定義 | 出力規模 | 修正上限 |
|--------|------|----------|----------|
| simple | 単一機能、1-2ファイル、外部依存なし | 50-100行、テスト5-8件 | 2回 |
| medium | 複数機能連携、2-3ファイル、軽量外部依存 | 100-200行、テスト8-15件 | 2回 |
| complex | 多機能、3ファイル以上、状態管理・永続化 | 150-300行/Phase、テスト10-20件 | Phase毎2回 |

complexタスクはPhase A/B/Cに分割し、Phase単位で承認を取る。

### 2.3 動的媒体選択（v0.4 由来）

v0.4で10媒体を定義したが、実装・検証可能な4媒体に絞った（v0.5の判断）。

| ID | 媒体 | 出力形式 | 判定キーワード |
|----|------|----------|---------------|
| cli | CLIツール | Python | ツール、スクリプト、自動化、Python |
| web | Webサイト/LP | HTML/CSS/JS | ホームページ、LP、サイト、HTML |
| doc | ドキュメント | Markdown | 企画書、レポート、マニュアル、手順書 |
| writing | 文章・創作 | テキスト | 小説、物語、記事、ブログ、脚本 |

### 2.4 専門家の固定割当（v0.4構想→v0.5で簡素化）

v0.4では13名の専門家を動的選出する構想だったが、選出ロジックの曖昧さが問題となり、媒体別に固定化した。

| 媒体 | Coder役 | Reviewer役 |
|------|---------|------------|
| cli | Pythonエンジニア | QAエンジニア |
| web | フロントエンドエンジニア | UXデザイナー |
| doc | テクニカルライター | 編集者 |
| writing | 小説家 | 文芸編集者 |

### 2.5 環境プロファイル（v0.5 由来）

Windows cp932エンコーディングエラーの実測から導入。cli媒体の場合、実装前にユーザーのOS・Pythonバージョン・利用可能パッケージを確認する。不明な場合はWindows / Python 3.12 / 標準ライブラリのみと仮定する。

### 2.6 出力量ガイドライン（v0.5 由来）

ツール呼び出し爆発（20回以上）の実測から導入。コードブロック1個あたり300行を上限とし、超える場合はファイルを分割する。

## 第3章 安全性（v0.6 由来）

### 3.1 セキュリティチェック

v0.6の有識者レビュー（Security Expert）により導入。Reviewer検証時に以下を確認する。

**禁止パターン:**
- eval(), exec(), compile() の使用（ast.literal_evalは許可）
- os.system(), subprocess の無制限使用
- __import__() による動的インポート
- pickle.loads() での信頼できないデータ読み込み

**機密情報チェック:**
- APIキー、トークン、パスワードのハードコード禁止
- .envファイルの内容の直接出力禁止
- 個人情報（メールアドレス、電話番号）の埋め込み禁止

**通信チェック:**
- 外部URLへの通信はユーザーの明示的要求がある場合のみ許可

### 3.2 安全制約

- 破壊的コマンド（rm -rf, format, shutdown等）を提案しない
- 外部サーバーへのデータ送信コードを書かない（明示的要求を除く）

## 第4章 ユーザー介入（v0.6 由来）

### 4.1 介入ポイント

v0.6の有識者レビュー（Product Manager）により導入。

- **complexタスク**: Step 1（計画）表示後にユーザー確認を挟む
- **simple/medium**: 確認なしで直行（速度優先）

### 4.2 介入の効果

ユーザーが計画段階で方向修正できるため、実装後の大幅な手戻りを防止する。

## 第5章 構造化ログ（v0.6 由来）

### 5.1 実行サマリー

v0.6のベンチマーク構想を、プロンプト出力として実現。毎回の応答末尾に以下を表示する。

📊 Synapse v1.0 Summary

COMPLEXITY: simple | medium | complex
MEDIA: cli | web | doc | writing
CODER_ROLE: （担当した専門家）
REVIEWER_ROLE: （担当した検証者）
FILES: （生成したファイル一覧）
ROUNDS: N（修正M回）
VERDICT: APPROVED | APPROVED（条件付き）
NOTE: （特記事項）

### 5.2 ログの活用

蓄積したサマリーを分析することで、媒体別の成功率、平均修正回数、頻出する問題パターンを把握し、プロンプト改善に活かす。これはv0.4で構想した自己進化ループのPhase 1に相当する。

## 第6章 プラットフォーム対応

### 6.1 実行環境の差異

| 機能 | GPTs (ChatGPT) | Gem (Gemini) | Claude | API自作エンジン |
|------|----------------|-------------|--------|----------------|
| コード実行 | Code Interpreter | NotebookLM連携 | Artifacts | Dockerサンドボックス |
| ファイル出力 | ダウンロード可 | なし | Artifacts | write_file |
| ナレッジ参照 | アップロード | NotebookLM | Projects | プロンプト注入 |
| テスト実行 | 自動実行可 | 不可（机上レビュー） | 一部可 | 自動実行可 |

### 6.2 プラットフォーム別の挙動調整

- **GPTs**: Code Interpreterでコード実行・テスト確認後にAPPROVED判定。ファイルはダウンロード可能な形で提供
- **Gem + NotebookLM**: コード実行不可の場合は机上レビュー。「ローカルで実行してテストしてください」と明記。ナレッジはNotebookLMから参照
- **Claude**: Artifacts機能でプレビュー表示。コード実行でテスト可能
- **API自作エンジン**: v0.3設計のDockerサンドボックスで完全自動実行

## 第7章 実測データ

### 7.1 API版（Anthropic Claude）での実績

| タスク | 複雑度 | 媒体 | ラウンド | コスト | 結果 |
|--------|--------|------|----------|--------|------|
| 電卓アプリ | simple | cli | 1 | $0.22 | APPROVED |
| じゃんけんゲーム | simple | cli | 1 | $0.29 | APPROVED |
| ローグライクダンジョン | complex | cli | 4 | $2.94 | 未承認（ツール呼び出し爆発） |

### 7.2 実測からの教訓

1. simpleタスクは1ラウンドで安定的にAPPROVED（コスト$0.2-0.3）
2. complexタスクはPhase分割なしだとコストが10倍以上に膨張する
3. 環境情報（OS、エンコーディング）の未注入が失敗の主因になりうる
4. ツール呼び出し回数の上限がないと際限なく試行を繰り返す

## 第8章 自己進化ループ（v0.4構想→v1.0で再定義）

### 8.1 現在の位置

v0.4で構想したPhase 1-4のうち、現在はPhase 1（人間が設計、AIがコード/コンテンツ生成）。

### 8.2 v1.0での実現方法

1. 構造化ログ（第5章）を蓄積する
2. ログを分析し、失敗パターンとプロンプト改善案をAIに提案させる
3. 人間が改善案を承認し、プロンプトを更新する
4. 更新後の実行結果を再びログに記録する

この反復がv0.4で描いた自己進化ループのPhase 2に相当する。

### 8.3 安全策

- 人間が各サイクルで必ず承認する（Human Sovereignty原則）
- 設計書の差分を明示し、変更点を透明にする
- コア設計原則（第1章 1.3）は人間のみ変更可能
- 各バージョンをGitで管理し、ロールバック可能

| 用語 | 定義 |
|------|------|
